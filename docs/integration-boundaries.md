# Границы ответственности: Next.js (сайт) vs внешние сервисы

Status: Normative  
Purpose: не дублировать бизнес-логику и интеграции, которые живут в отдельном backend (Python / RestoMind и т.д.).

Связано с: [CODEBASE.md](CODEBASE.md), [deployment-runbook.md](deployment-runbook.md).

---

## Что остаётся в этом репозитории (Next.js)

- Публичный сайт, меню из Supabase, гостевой заказ через server actions и `submitOrder`.
- Админка (KDS, стоп-лист, дашборд), Realtime на `orders` / `menu_items`.
- SEO, лендинг из `TenantPublicConfig`, Telegram-уведомления о заказе (опционально).

---

## Чего не делать в проекте сайта (без дублирования)

Ниже задачи **намеренно не реализуются** в Next.js: их зона — отдельный Python-сервис (webhooks Meta, AI, сложный прайсинг упаковки, загрузка номенклатуры iiko) или общая БД.

| Зона | Почему не в Next.js |
|------|---------------------|
| **WhatsApp webhooks** | Обработка входящих, дедупликация, фоновые задачи — в Python; webhook в Meta указывает на тот сервис, не на Vercel. |
| **AI-диалог и заказы через LLM** | Контракты ответа (`AIBrainResponse`), системные промпты и оркестрация — в Python; сайт не дублирует OpenAI-логику заказов. |
| **Сложный расчёт упаковки / fee lines** | Логика вроде наценок за контейнеры, правил «табак vs казан» и т.п. — в Python (`order_logic` и аналоги) или в хранимой процедуре БД; Next.js передаёт согласованный с текущей схемой состав заказа, без второго движка цен. |
| **Синхронизация номенклатуры iiko** | Не второй загрузчик меню в Next.js; источник истины для загрузки — скрипты/сервис на Python; приложение **читает** уже заполненные таблицы в Supabase. |

При добавлении фич проверяйте этот список, чтобы не плодить расхождения с backend.
